Detailed Flow Description

Step 1: User Interaction

User enters repository owner and name in the frontend form and submits the request.

// Frontend form submission document.getElementById('repoForm').addEventListener('submit', async function(e) { e.preventDefault(); const owner = document.getElementById('owner').value; const repo = document.getElementById('repo').value; await fetchRepoStats(owner, repo); });

Step 2: API Request to PR Stats Server

Frontend makes an AJAX request to the PR Stats Server endpoint.

// Frontend API call async function fetchRepoStats(owner, repo) { showLoading(); try { const response = await fetch(`/api/repo-stats?owner=${owner}&repo=${repo}`); const stats = await response.json(); updateUI(stats); } catch (error) { showError(error); } finally { hideLoading(); } }

Step 3: PR Stats Server Processing

The PR Stats Server receives the request and spawns a child process to run the repo-stats.js script.

// repo-stats-server.js app.get('/api/repo-stats', async (req, res) => { const { owner, repo } = req.query; try { const stats = await getRepoStats(owner, repo); res.json(stats); } catch (error) { res.status(500).json({ error: error.message }); } });

Step 4: GitHub Data Collection

The repo-stats.js script uses MCP tools to fetch data from GitHub API.

// repo-stats.js async function getRepoStats(owner, repo) { // 1. Get all PRs in the repository const allPRs = await mcp0_search_pull_requests({ query: `repo:${owner}/${repo} is:pr`, perPage: 100 }); // 2. Get all commits in the repository const commits = await mcp0_list_commits({ owner, repo, perPage: 100 }); // Process data and generate statistics... }

Step 5: Data Processing and Analysis

The script processes the raw data to generate various statistics.

  • Contributor Statistics: Commits, PRs, and activity per contributor
  • File Statistics: Most changed files and their PR counts
  • Defect Areas: Directories with most DEF PRs (PRs with titles starting with "DEF")
  • PR Guidelines: PRs violating size limits, open too long, slow reviews, unresolved comments

Step 6: Response to Frontend

The processed statistics are sent back to the frontend.

Step 7: UI Update

The frontend updates the UI with charts, tables, and visualizations.

// Frontend UI update function updateUI(stats) { const { contributorStats, fileStats, defectStats, prGuidelines, commitStats } = stats; updateSummaryStats({ ... }); updateContributorStats(contributorStats); updateFileStats(fileStats); updateDefectStats(defectStats); updateGuidelinesStats(prGuidelines); }

PR Guidelines Compliance Analysis

The tool analyzes PRs for compliance with the following guidelines:

  1. Size Limit: PRs should not exceed 300 lines of changes
  2. Open Time: PRs should not remain open for more than 4 hours
  3. Review Time: First review should happen within 24 hours
  4. Unresolved Comments: PRs should not be merged with unresolved review comments

For unresolved comments detection, the tool:

  1. Fetches PR review comments using mcp0_get_pull_request_comments
  2. Fetches PR reviews using mcp0_get_pull_request_reviews
  3. Identifies reviewers whose last review requested changes but never approved the PR
  4. Collects detailed comment information including body, author, creation date, file path, and line number

Defect Areas Analysis

The tool identifies defect-prone areas by:

  1. Identifying PRs with titles starting with "DEF" (case-insensitive)
  2. Tracking which directories are affected by these DEF PRs
  3. Counting each DEF PR once per directory to avoid double counting
  4. Visualizing the results in bar charts, pie charts, and tables

This helps identify which parts of the codebase have the most defect fixes, potentially indicating areas that need more attention, code reviews, or refactoring.
